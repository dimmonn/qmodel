This document is based on `com.research.qmodel.controllers.BaseGitQueryController` and describes the REST endpoints exposed under the base path:

`/api/v1/metrics`

== Overview

The controller primarily exposes repository import and analysis endpoints. Most routes are `GET` requests that trigger data collection, persistence, or SZZ tracing side effects.

== Common Request Model

Several batch endpoints accept a JSON array of `Project` objects.

[source,json]
----
[
  {
    "projectOwner": "ansible",
    "projectName": "ansible"
  },
  {
    "projectOwner": "google",
    "projectName": "guava"
  }
]
----

`Project` fields:

[cols="1,3",options="header"]
|===
| Field | Description

| `projectOwner`
| GitHub owner or organization name.

| `projectName`
| GitHub repository name.
|===

== Endpoint Details

=== `GET /repos`
Returns the list of projects currently stored in the persistence layer.

=== GET /repos/<owner>/<repo>/actions
Fetches Actions check-runs for the repository and stores the result.

Notes:

* This route is side-effecting despite using `GET`.
* It is intended for repository-level Actions metadata.

=== GET /repos/<owner>/<repo>/commits/actions
Loads check-runs for each commit already present in the local database.

Flow:

1. Load all commits for the project from the repository.
2. For each commit SHA, call GitHub check-runs.
3. Persist the returned batch with the commit SHA.

=== GET /repos/<owner>/<repo>/retrieveFixingCommits
Calls `findAllBugsFixingCommits(repo, owner, 2)` and returns the SHAs of candidate fixing commits.

=== GET /repos/<owner>/<repo>/<id>/retrieveBugIntroducingCommits
Calls `findBugIntroducingCommits(owner, repo, id, 2)` and returns the candidate bug-introducing commits linked to the issue.

=== GET /repos/<owner>/<repo>/retrieveBugIntroducingCommits?depth=<depth>
Runs `traceCommitsToOrigin(owner, repo, depth)` to traverse commit history back to origin with the requested depth.

=== GET /repos/<owner>/<repo>/commits
Streams the complete commit history and persists it in batches of `app.page_size`.

=== GET /repos/<owner>/<repo>/commits/<sha>
Imports a single commit by SHA and persists the resulting graph snapshot.

=== GET /repos/<owner>/<repo>/forked
Retrieves forks for the repository, computes fork-aware graph data, persists it, and returns the discovered fork full names.

=== GET /repos/<owner>/<repo>/forks
Clones the repository to a local working directory and inspects forked commits from the local checkout.

=== GET /repos/<owner>/<repo>/pulls
Fetches closed pull requests and stores them in the database.

=== `GET /repos/pulls`
Batch version of the pull-request import endpoint. Accepts a JSON array of `Project` objects in the request body.

=== GET /repos/<owner>/<repo>/issues
Fetches closed issues for a project and returns the number of issues retrieved.

=== DELETE /repos/<owner>/<repo>/issues
Deletes all issues already stored for the project.

=== `GET /repos/fixtime`
Batch issue-import endpoint for multiple projects. Accepts a JSON array of `Project` objects.

=== `GET /repos/ag`
Batch graph-import endpoint for multiple projects. Accepts a JSON array of `Project` objects.

  --url http://localhost:8082/api/v1/metrics/repos/ansible/ansible/commits

curl --request GET \
  --url http://localhost:8082/api/v1/metrics/repos/ansible/ansible/issues

curl --request DELETE \
  --url http://localhost:8082/api/v1/metrics/repos/ansible/ansible/issues

curl --request GET \
  --url http://localhost:8082/api/v1/metrics/repos/ansible/ansible/3/retrieveBugIntroducingCommits
----

== Notes

* The controller uses `GET` for several import operations to keep the API simple, even when the endpoint mutates state.
* Some endpoints depend on the current database state, so results may change as more commits, issues, pull requests, or fork data are imported.
* The controller also relies on internal services such as `BasicQueryService`, `BasicBugFinder`, `CommitRepository`, and `DataPersistance` for the actual work.
